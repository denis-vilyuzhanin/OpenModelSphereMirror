Comparing and integrating models
 

The integration is the interactive process in which two models are compared and can be synchronized.

It is the process of updating a model using another model. This feature uses semantic information from models of the same type. Here is a list of all the model types that can be integrated.
 

Model types

  • Common item model
  • Conceptual data model
  • Relational data model
  • Domain model
  • Operation library
  • Database
  • Process model
  • Class model
  • Package


To integrate models:

  1. Choose the model to be integrated
  2. Select Tools > Compare/Integrate... in the main menu.
  3. A window appears.

Specification: With a data model, the integration will create hierarchical links if you compare subs and supermodels.
 

Options

Use as identifier: the integration process uses object names or physical names in the matching process and the identifier is displayed in the integration window.

Scope: the process applies a filter to the concepts and descriptors that are compared. This list is based on a compatible set of concepts with regards to the type and the target system, if any, of the selected models. It is possible to save and reload (.scp) integration scope parameters.

Match external objects using: an external object is one that is not included in the two models selected for integration. For example, when integrating two data models, it is likely that columns will refer to domains that are not part of a specific domain model, the domain is an external object.

  • [Identifier]: the integration reports a difference if both external references point to two different objects not having the same name or physical name. In a data model example, if two columns’ data type is linked to different domains having the same name, no difference will be reported for the type. Of course, if domains have different names then a difference will be detected.

    •  
  • [Reference]: the integration reports a difference if the external object reference does not point to the exact same object (compare by reference). In our data model example, if the two columns’ data type is linked to different domains then a difference will automatically be detected without regards to whether the name and physical name are the same.

    •  


Process

The integration process uses the hierarchical links to associate objects and the name or the physical name according to the selected value.

The upper portion of the interface window displays side-by-side information concerning the compared models. You will notice that the tree structure resembles the explorer window's view.

BLUE The object is only in model 1
GREEN The object is only in model 2
RED Objects are different
BLACK Objects are identical

The lower portion will display differences, if any, between the object properties.
 

Object actions

After comparing the two models and displaying the differences, the user can modify the models in a selective manner, and this in any direction. However, it is important to specify that all actions are applied hierarchically on the object selection. Here are the actions offered:

  • Merge: Source model objects are added to the target model to form a superset of all model components. (Add/modify)

    •  
  • Replace: The target model information is replaced by the source model information. Any additional information in the target is deleted.

    •  
  • None: No action will be applied on the object and its descendants.

    •  
  • Associate: Link two objects so they are considered equivalents and will be compared.

    •  
  • Break Association: Unlink two objects so they are not considered equivalents and will not be compared.

    •  
There are also actions available on object property differences:
  • Modify: Modify the value of the source or the target object’s properties.

    •  
  • Display Values: Offers a larger size display of source and target values.

    •  


Report

A report is always produced at the end of an integration process. This report gives information on the two models that were compared, the date and time, the list of differences and the modifications applied (if there are any). For impact analysis purposes, this same report can also be produced before the integration.
